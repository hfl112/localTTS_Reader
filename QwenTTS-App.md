# QwenTTS-App 设计文档 (Phase 1)

## 1. 项目愿景
将 Qwen3-TTS 1.7B 极致的听感转化为一个随时可用的 macOS 原生菜单栏工具。不再局限于浏览器插件，实现全系统的“所见即所读”。

## 2. 软件架构 (Modular Architecture)

### **UI 层 (Main UI)**
*   **rumps**: 极简的 macOS 菜单栏框架。
*   **pyperclip**: 用于快速获取剪贴板内容。
*   **功能**:
    *   `朗读剪贴板`: 一键触发，自动处理当前选中的文字。
    *   `恢复播放`: 利用持久化存储，回到上次关闭前的位置。
    *   `快捷设置`: 在菜单中直接切换音色 (Serena/Ryan)。

### **核心层 (Core Services)**
1.  **`backend.py` (进程级隔离中心)**:
    *   **架构分化**: 采用 `multiprocessing` 实现推理引擎的物理隔离。
    *   **SharedState**: 跨进程状态机，实时同步 VRAM 占用、缓冲区水位及引擎运行状态。
    *   **Cruise Mode**: 8s-15s 动态缓冲区巡航，维持热量平衡。
    *   **资源回收**: 10 分钟空闲 VRAM 自动卸载，启动时自动清理残留进程。
2.  **`processor.py`**:
    *   **Markdown 脱敏**: 针对 Obsidian 深度优化，自动过滤 YAML 及图片语法。
    *   **会话 LRU 缓存**: 最近 10 次朗读 MD5 哈希缓存，实现“复读零负载”。
3.  **`tts_engine.py`**:
    *   **16kHz 性能下沉**: 在 MLX 层直接输出降采样单声道数据，节省 33% 开销。
4.  **`player.py` (极致驱动)**:
    *   **Zero-Logic Callback**: 纯内存拷贝回调，彻底免疫 GIL 锁竞争。
    *   **智能监控**: 仅在活跃任务时触发性能日志，IDLE 状态彻底静默。

## 3. 模块化生态架构 (Modular Ecosystem)

目前的系统采用“**物理独立，逻辑合体**”的架构，这是一种面向未来的设计选择：

### **A. 为什么 App 与插件保持物理分离？**
1.  **技术栈解耦**：App 基于 Python (MLX/算力)，插件基于 TypeScript (浏览器/抓取)。分离确保了各组件可以独立升级而不互相干扰。
2.  **基站模式 (Hub & Spoke)**：
    *   **App = 基站**：负责最沉重的模型加载、GPU 推理和本地 PCM 播放。
    *   **Extension = 遥控器**：负责在网页端抓取文本并远程投喂给 App。
3.  **多端扩展性**：这种“基站模式”意味着未来可以轻松接入其他客户端（如 iPad 远程控制器、手机 PWA 或 PDF 阅读器），所有设备共享同一份驻留内存的模型。

### **B. 协作流程**
*   你在浏览器中右键点击 -> 插件通过本地 8001 端口发送文本。
*   App 接收文本 -> 自动重置进度 -> 瞬间在 Mac 音箱出声。
*   App 菜单栏同步显示来自浏览器的阅读进度。
